# v0.4.4 - Fix React Preamble Detection Error

## Date
2026-03-09

## Description
Fixed error: "@vitejs/plugin-react can't detect preamble. Something is wrong." that was appearing in browser console when loading the welcome page.

## Changes Made

### vite.config.ts
- Removed conflicting `esbuild.jsx` configuration
- The `esbuild: { jsx: 'automatic' }` setting was conflicting with `@vitejs/plugin-react` plugin
- Updated React Compiler configuration to use proper array format with target option

### resources/views/app.blade.php
- **CRITICAL FIX**: Removed duplicate entry point for page components
- Changed from: `@vite(['resources/js/app.tsx', "resources/js/pages/{$page['component']}.tsx"])`
- Changed to: `@vite(['resources/js/app.tsx'])`
- Inertia.js handles page component loading dynamically via `import.meta.glob('./pages/**/*.tsx')`
- Loading page components directly in blade causes React Fast Refresh preamble conflicts

## Issue Details
**Symptoms:**
- Browser console showed repeated errors:
  - `@vitejs/plugin-react can't detect preamble. Something is wrong.`
  - Error appeared on welcome.tsx:9:29
  - Error appeared both on page load and in Inertia.js promises

**Root Cause:**
Two separate issues were causing the error:

1. **Vite Configuration Conflict:**
   - `@vitejs/plugin-react` plugin handles JSX transformation and React Fast Refresh
   - `esbuild: { jsx: 'automatic' }` also tried to handle JSX transformation
   - These two configurations conflicted

2. **Multiple Entry Points in Blade Template:**
   - Blade template was loading both `app.tsx` AND individual page components (e.g., `welcome.tsx`)
   - Page components were being loaded as separate Vite entry points
   - Each entry point tried to inject React Fast Refresh preamble
   - Since `app.tsx` already initializes Inertia and loads all pages via `import.meta.glob`, loading pages directly created conflicts
   - The React Fast Refresh preamble detection failed because of these conflicting entry points

**Resolution:**
1. Removed the `esbuild.jsx` configuration from `vite.config.ts`
2. Removed the dynamic page component loading from `app.blade.php`
3. Updated React Compiler plugin configuration to proper array format with target option

Inertia.js architecture:
- `app.tsx` is the single entry point that initializes the Inertia app
- Page components are loaded dynamically via `resolvePageComponent()` which uses `import.meta.glob('./pages/**/*.tsx')`
- This is the correct pattern for Inertia + Vite applications
- Loading page components separately in blade breaks this pattern

## Technical Notes
- The `@vitejs/plugin-react` plugin uses Babel for JSX transformation
- It injects preamble code for React Fast Refresh at the top of entry point files
- Having both esbuild and React plugin handle JSX breaks preamble injection
- Multiple Vite entry points for the same React app cause Fast Refresh conflicts
- The plugin configuration with `babel-plugin-react-compiler` uses proper array syntax

## Testing
- Development mode: React Fast Refresh works properly
- Production build: No preamble detection errors
- Welcome page loads without console errors
