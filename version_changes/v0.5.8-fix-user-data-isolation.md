# Version 0.5.8 - Fix User Data Isolation

## Release Date
2026-03-09

## Problem
Data user lain dapat diakses ketika user berbeda login. Saat ini jika ada user lain yang login dengan akun berbeda, data user pertama masih tampil. Setiap user harus memiliki data terpisah (1 user = 1 data).

## Root Cause
Masalah ini disebabkan oleh:
1. **Browser Caching**: Browser menyimpan cache data dari user sebelumnya
2. **Inertia Client-side Caching**: Inertia menyimpan cache props yang bisa muncul kembali setelah user logout dan login dengan akun berbeda
3. **localStorage Persistence**: Data tersimpan di localStorage tidak dibersihkan saat user berganti akun

## Solution
Melakukan beberapa perubahan untuk memastikan isolasi data antar user:

### 1. Backend Changes

#### `app/Http/Middleware/HandleInertiaRequests.php`
- **Modified `version()` method**: Menambahkan user ID ke dalam version string untuk memaksa cache invalidation saat user berganti
  ```php
  public function version(Request $request): ?string
  {
      $userId = $request->user()?->id;
      return parent::version($request) . ($userId ? "-user-{$userId}" : '');
  }
  ```

- **Added `userId` to shared props**: Mengirim user ID ke frontend untuk deteksi perubahan user
  ```php
  'userId' => $request->user()?->id,
  ```

#### `app/Http/Middleware/PreventCaching.php` (NEW)
- Created new middleware untuk mencegah browser caching
- Menambahkan headers: `Cache-Control`, `Pragma`, dan `Expires`

#### `bootstrap/app.php`
- Registered `PreventCaching` middleware ke web middleware group

### 2. Frontend Changes

#### `resources/js/components/app-shell.tsx`
- Added user change detection menggunakan `useRef` dan `useEffect`
- Automatic localStorage cleanup saat user berganti
- Automatic page reload untuk memastikan state bersih
  ```typescript
  const currentUserId = usePage().props.userId as number | undefined;
  const previousUserId = useRef<number | undefined>(currentUserId);

  useEffect(() => {
    if (previousUserId.current !== undefined && previousUserId.current !== currentUserId) {
      localStorage.clear();
      window.location.reload();
    }
    previousUserId.current = currentUserId;
  }, [currentUserId]);
  ```

## Technical Details

### How It Works
1. **Version-based Cache Invalidation**: Setiap user memiliki version string yang unik (misal: `version-user-1`, `version-user-2`)
2. **User ID Tracking**: Frontend memantau perubahan user ID dari Inertia props
3. **Automatic Cleanup**: Saat user ID berubah, localStorage di-clear dan halaman di-reload
4. **No Browser Caching**: Semua response memiliki headers yang mencegah caching

### Data Isolation Flow
1. User A login → Inertia version: `v1-user-1`
2. User A logout, User B login → Inertia version: `v1-user-2`
3. Frontend mendeteksi perubahan user ID
4. localStorage dibersihkan
5. Halaman di-reload dengan data User B

## Testing
Untuk menguji perubahan ini:
1. Login dengan User A
2. Lakukan beberapa scan makanan
3. Logout
4. Login dengan User B
5. Pastikan data User A tidak muncul
6. Lakukan scan baru untuk User B
7. Pastikan data User A dan User B terpisah

## Impact
- ✅ Data user sekarang benar-benar terisolasi
- ✅ Browser caching di-disabled untuk aplikasi Inertia
- ✅ localStorage otomatis dibersihkan saat user berganti
- ✅ Tidak ada kebocoran data antar user

## Notes
- Backend controllers sudah memiliki proper user filtering dengan `Auth::id()` checks
- Masalah utama adalah pada caching layer, bukan pada authorization
- Perubahan ini tidak mempengaruhi performa secara signifikan
- Version-based caching adalah pattern yang direkomendasikan oleh Inertia.js untuk user-specific data
