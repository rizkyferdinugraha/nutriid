# v0.2.1 - Separate Nutrition Profile Table

## Date
2026-03-08

## Summary
Separated nutrition profile data from users table into a dedicated `nutrition_profiles` table. This resolves validation conflicts and provides a cleaner separation of concerns between user authentication/profile data and nutrition-specific data.

## Problem Solved

**Issue**: Nutrition profile form was sharing the same endpoint as profile information update, which required name and email fields even though the nutrition form only needed age, weight, height, and gender. This caused validation errors and prevented saving nutrition data.

**Solution**: Created a separate `nutrition_profiles` table with its own dedicated controller, routes, and page.

## Changes Made

### Database Changes
- **New Migration**: `2026_03_08_015302_create_nutrition_profiles_table.php`
  - Created separate table for nutrition profile data
  - Fields: `user_id` (foreign key), `age`, `weight`, `height`, `gender`
  - Unique constraint on `user_id` (one profile per user)
  - Cascade delete when user is deleted

### Backend Changes

#### Models
- **New**: `NutritionProfile.php` model
  - Fillable fields: `age`, `weight`, `height`, `gender`
  - Type casts for proper data handling
  - `user()` relationship to User model
  - `getOrCreateForUser()` helper method - returns null if no profile exists
  - `updateOrCreateForUser()` helper method - creates or updates profile with data

#### Controllers
- **New**: `NutritionProfileController.php`
  - `edit()` - Show nutrition profile settings page
  - `update()` - Update nutrition profile data
  - Routes to separate page: `/settings/nutrition-profile`

#### Controllers (Updated)
- **FoodScanController.php**
  - Updated `dailySummary()` to use `NutritionProfile` model instead of `User` model
  - Now reads nutrition data from `nutrition_profiles` table

- **ProfileController.php**
  - Removed `updateNutritionProfile()` method (no longer needed)
  - Simplified to handle only name and email updates

#### Requests
- **New**: `NutritionProfileUpdateRequest.php`
  - Validation rules for nutrition profile fields
  - All fields are optional (nullable)
  - Age: 1-120 years
  - Weight: 20-300 kg
  - Height: 100-250 cm
  - Gender: male/female

### Routes
- **New**: `settings/nutrition-profile`
  - GET `/settings/nutrition-profile` - Show nutrition profile form
  - PATCH `/settings/nutrition-profile` - Update nutrition profile
- **Removed**: `/settings/profile/nutrition` - No longer needed

### Frontend Changes

#### Pages
- **New**: `nutrition-profile.tsx`
  - Dedicated page for nutrition profile settings
  - Standalone form with age, weight, height, gender fields
  - Uses `NutritionProfileController.update.form()` action
  - Proper TypeScript types for profile data
  - Breadcrumb navigation
  - Success message display

- **Updated**: `profile.tsx`
  - Removed entire nutrition profile form section
  - Added link button to navigate to nutrition profile page
  - Now only handles name and email updates
  - Cleaner, more focused user experience

#### Components
- **Updated**: `user-menu-content.tsx`
  - Added "Nutrition Profile" menu item with HeartPulse icon
  - Direct access to nutrition profile page from user menu
  - Positioned above Settings for better discoverability

- **No changes**: daily-summary.tsx and nutrition-info-modal.tsx continue to work correctly

## Benefits

### Better Separation of Concerns
- User authentication/profile data separated from nutrition-specific data
- Each controller has a single, focused responsibility
- Cleaner codebase with less coupling

### Improved User Experience
- Dedicated page for nutrition profile settings
- No validation conflicts between profile and nutrition data
- Clear navigation with proper breadcrumbs

### Data Integrity
- One-to-one relationship between users and nutrition profiles
- Cascade delete ensures data consistency
- Nullable fields allow users to provide data gradually

### Maintainability
- Easier to extend nutrition features without affecting user management
- Independent validation rules
- Simpler testing scenarios

## Migration Notes

- Old data in `users` table (age, weight, height, gender fields from previous migration) is still present but no longer used
- New users will automatically have a `nutrition_profiles` record created when they first visit the page
- Existing users will get a new `nutrition_profiles` record when they first save their nutrition data

## User Flow

1. User goes to Settings → Profile
2. Clicks "Manage Nutrition Profile" button
3. Redirected to `/settings/nutrition-profile`
4. Fills in personal data (age, weight, height, gender)
5. Clicks "Simpan Profil"
6. Data saved to `nutrition_profiles` table
7. Dashboard shows personalized progress bars
8. Clicking nutrition cards opens modal with personalized targets
